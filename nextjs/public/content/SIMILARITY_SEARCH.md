# 유사도 검색 원리

## 개요

CL Embed는 **텍스트 임베딩(Text Embedding)** 과 **코사인 유사도(Cosine Similarity)** 를 활용하여 입력 텍스트와 가장 의미적으로 유사한 카테고리를 찾습니다. 단순 키워드 매칭이 아닌, 텍스트의 의미를 벡터 공간에서 비교하는 방식입니다.

## 핵심 기술

### 텍스트 임베딩

텍스트 임베딩은 문장이나 단어를 고정된 길이의 실수 벡터로 변환하는 기술입니다. 의미적으로 유사한 텍스트는 벡터 공간에서 가까운 위치에 배치됩니다.

- **모델**: `bge-m3:latest` (Ollama 기반 로컬 임베딩)
- **차원**: 각 텍스트는 1024차원 벡터로 표현됩니다

### pgvector

PostgreSQL의 `pgvector` 확장을 사용하여 벡터 데이터를 관계형 데이터베이스에 저장하고 검색합니다. pgvector는 벡터 간 코사인 유사도 연산을 위한 인덱싱과 쿼리 연산자를 제공합니다.

### 코사인 유사도

코사인 유사도는 두 벡터 간의 방향 유사성을 측정합니다. 값은 -1에서 1 사이이며, 1에 가까울수록 유사도가 높습니다.

```
cosine_similarity(A, B) = (A · B) / (|A| × |B|)
```

- `A · B`: 두 벡터의 내적(dot product)
- `|A|`, `|B|`: 각 벡터의 크기(norm)

## 데이터 파이프라인

### 1. 카테고리 준비

통합 카테고리 체계를 기준으로 한국어 카테고리명이 DB에 저장됩니다.

### 2. 번역

한국어 카테고리명을 Ollama 로컬 LLM을 통해 영어, 중국어로 번역합니다.

```
한국어 원문 → [Ollama LLM] → 영어 번역문
한국어 원문 → [Ollama LLM] → 중국어 번역문
```

### 3. 임베딩 생성

각 언어별 번역문을 임베딩 모델로 벡터화하여 저장합니다.

```
번역문 → [Embedding Model] → 벡터 → pgvector 저장
```

언어별로 독립적인 임베딩을 생성하므로, 동일한 카테고리라도 언어별로 다른 벡터 공간에 저장됩니다.

## 검색 처리 흐름

### Step 1: 입력 텍스트 수신

사용자가 입력한 검색어(상품명, 설명 등)를 수신합니다.

### Step 2: 캐시 확인

`search_logs` 테이블에서 동일한 검색어 이력을 조회합니다. 이전에 처리된 적 있는 검색어라면 저장된 벡터를 재사용하여 응답 속도를 단축합니다.

### Step 3: 임베딩 생성

새로운 검색어인 경우, 입력 텍스트를 임베딩 모델로 벡터화합니다. 생성된 벡터는 향후 캐시 히트를 위해 `search_logs`에 저장됩니다.

### Step 4: 코사인 유사도 계산

입력 벡터와 선택된 타겟 언어의 모든 카테고리 벡터 간 코사인 유사도를 계산합니다. PostgreSQL의 `<=>` 연산자를 사용하여 효율적으로 처리됩니다.

### Step 5: 결과 정렬 및 반환

코사인 유사도가 높은 순으로 정렬하여 상위 N개 결과를 반환합니다. 각 결과에는 카테고리명, 유사도 점수, 카테고리 코드가 포함됩니다.

## 성능

| 시나리오 | 응답 시간 |
|---|---|
| 캐시 히트 (동일 키워드) | 100ms 이하 |
| 신규 키워드 (임베딩 + 검색) | 1.5초 이하 |

> 실제 응답 시간은 실행 환경에 따라 차이가 있을 수 있습니다.
